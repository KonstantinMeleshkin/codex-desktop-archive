# Release Policy

## When A Release Is Created

A new GitHub Release is created only when the candidate manifest differs from `manifest/latest.json` by artifact identity.

Release-triggering changes include:

- macOS app version changed
- macOS app build changed
- macOS SHA-256 changed
- macOS size changed

## When A Release Is Skipped

The workflow exits successfully without creating a release when all artifact identities match the latest manifest.

Before it exits, the workflow verifies the existing GitHub Release against `manifest/latest.json`, including release assets, attached manifest, release notes, and tag target. A mismatch fails the workflow.

This avoids daily duplicate releases without allowing silent release-asset drift.

## Naming

Preferred tag:

```text
desktop-v<app-version>
```

Preferred macOS asset:

```text
Codex-Desktop-<app-version>-macos.dmg
```

## Immutability

Existing release assets must not be silently replaced.

The workflow must:

- avoid `--clobber`
- validate the release manifest before publication
- re-inspect the macOS DMG in the protected publish job before creating or reusing a release
- stage a draft release before publishing it
- verify the uploaded release assets against the manifest
- exit successfully if an existing release has identical verified assets, manifest, release notes, and tag target
- publish an existing draft release only after those same checks pass
- fail if an existing release has a different tag target, different assets, missing manifest, or unexpected assets
- commit `manifest/latest.json` only after the release has been created and verified

## Backfills

Historical backfills must be labeled according to evidence:

- `workflow-observed` when generated by this workflow
- `historical-observed` when matching a locally preserved file or historical GitHub asset
- `partial` when app-level verification is unavailable

Do not claim exact historical OpenAI bytes unless OpenAI or another independent source published the historical hash.
